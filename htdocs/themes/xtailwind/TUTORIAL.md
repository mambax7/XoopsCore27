# Building a Tailwind CSS Theme for XOOPS

A practical guide to creating a modern Tailwind CSS theme for XOOPS 2.5, including how to port an existing Bootstrap theme.

**Who this is for:** developers who know their way around XOOPS themes and are comfortable with Tailwind's utility-first approach. No prior Tailwind experience required.

**What you'll build:** a working theme with DaisyUI components, light/dark mode, and a live theme switcher — the same architecture used in the xtailwind reference theme.

---

## Part 1 — Creating a Tailwind Theme From Scratch

### Prerequisites

- XOOPS 2.5.11 or later
- PHP 8.2 or later
- **Node.js 18+** (only needed at build time, not at runtime)
- A text editor you like

### Step 1 — Directory structure

XOOPS themes live in `htdocs/themes/<yourtheme>/`. Create this minimal structure:

```
themes/mytheme/
├── theme.ini              # XOOPS theme metadata
├── theme.tpl              # Main page template
├── theme_autorun.php      # PHP bootstrap (form renderer, Smarty vars)
├── package.json           # npm dependencies and build scripts
├── tailwind.config.js     # Tailwind + DaisyUI configuration
├── css/
│   ├── input.css          # Source CSS (@tailwind directives)
│   └── styles.css         # Compiled output (generated by build)
├── tpl/
│   ├── nav-menu.tpl       # Navigation
│   ├── content-zone.tpl   # Main content area
│   ├── leftBlock.tpl      # Left sidebar blocks
│   ├── rightBlock.tpl     # Right sidebar blocks
│   └── blockContent.tpl   # Block wrapper
└── language/english/main.php  # Translatable strings
```

Also add `index.php` guards in each subdirectory (copy the standard XOOPS header).

### Step 2 — theme.ini

XOOPS reads this to display your theme in the admin panel:

```ini
[Theme]
Name="MyTheme"
Description="A Tailwind CSS theme for XOOPS 2.5"
Version="1.0.0"
Author="Your Name"
W3C="HTML5"
Licence="MIT/GPL v3"
thumbnail="thumbnail.png"
screenshot="screenshot.png"
```

### Step 3 — Install Tailwind + DaisyUI

From inside your theme directory:

```bash
cd themes/mytheme
npm init -y
npm install -D tailwindcss@^3.4.0 @tailwindcss/typography@^0.5.10 daisyui@^4.12.0
npx tailwindcss init
```

This creates `package.json`, installs dependencies into `node_modules/`, and generates a basic `tailwind.config.js`.

Add build scripts to `package.json`:

```json
{
    "scripts": {
        "build": "tailwindcss -i ./css/input.css -o ./css/styles.css --minify",
        "watch": "tailwindcss -i ./css/input.css -o ./css/styles.css --watch"
    }
}
```

### Step 4 — tailwind.config.js

Replace the generated config with:

```js
/** @type {import('tailwindcss').Config} */
module.exports = {
    content: [
        './theme.tpl',
        './tpl/**/*.tpl',
        './modules/**/*.tpl',
    ],
    theme: {
        extend: {},
    },
    plugins: [
        require('@tailwindcss/typography'),
        require('daisyui'),
    ],
    daisyui: {
        themes: ['light', 'dark', 'cupcake', 'corporate', 'synthwave'],
        darkTheme: 'dark',
    },
};
```

**What matters here:**

- **`content`** — the JIT compiler scans these files for class names. Only classes that appear get included in the output CSS. This keeps the final file small.
- **`plugins`** — `@tailwindcss/typography` provides the `prose` class for styling rich-text content (like XOOPS article output). `daisyui` gives you component classes (`btn`, `card`, `navbar`) and pre-built themes.
- **`daisyui.themes`** — pick which themes to bundle. Each theme adds ~2–3 KB. Start with a few; you can add more later.

### Step 5 — css/input.css

The source CSS file with Tailwind directives:

```css
@tailwind base;
@tailwind components;
@tailwind utilities;

/* Hides elements with the Alpine.js x-cloak attribute until Alpine initializes */
[x-cloak] { display: none !important; }

@layer components {
    /* Your custom component classes go here */
    .block-content ul { list-style: none; padding: 0; margin: 0; }
    .block-content li { display: block; }
    .block-content a {
        display: block;
        padding: 0.375rem 0.5rem;
        border-radius: 0.375rem;
    }
    .block-content a:hover {
        background-color: hsl(var(--b3));
    }
}
```

`hsl(var(--b3))` uses DaisyUI's CSS variables — it automatically picks the current theme's color, so your custom styles stay theme-aware.

### Step 6 — theme_autorun.php

This file runs on every page load before templates render. Use it to set the XOOPS form renderer and pass data to Smarty:

```php
<?php
xoops_load('XoopsFormRendererBootstrap5');
XoopsFormRenderer::getInstance()->set(new XoopsFormRendererBootstrap5());

global $xoopsTpl;
if (!empty($xoopsTpl)) {
    $xoopsTpl->addConfigDir(__DIR__);
    $xoopsTpl->assign('mythemeThemes', [
        ['name' => 'light', 'label' => 'Light', 'mode' => 'light'],
        ['name' => 'dark',  'label' => 'Dark',  'mode' => 'dark'],
        // Match this list to the `themes` array in tailwind.config.js
    ]);
}
```

**Why the Bootstrap 5 form renderer?** XOOPS core doesn't ship a Tailwind-specific form renderer. The Bootstrap 5 one outputs class names like `.form-control` and `.btn` that DaisyUI also defines — so forms render reasonably well without extra work.

### Step 7 — theme.tpl

The main page template:

```smarty
<!doctype html>
<html lang="<{$xoops_langcode}>" dir="<{$xoops_text_direction|default:'ltr'}>" data-theme="light">
<head>
<{assign var=theme_name value=$xoTheme->folderName}>
    <meta charset="<{$xoops_charset}>">
    <meta name="viewport" content="width=device-width, initial-scale=1">

    <link rel="stylesheet" type="text/css" href="<{xoImgUrl 'css/styles.css'}>">

    <{* FOUC prevention: apply saved theme before paint *}>
    <script>
    (function() {
        const stored = localStorage.getItem('mytheme-theme');
        const prefersDark = window.matchMedia('(prefers-color-scheme: dark)').matches;
        document.documentElement.setAttribute('data-theme', stored || (prefersDark ? 'dark' : 'light'));
    })();
    </script>

    <script src="<{$xoops_url}>/browse.php?Frameworks/jquery/jquery.js"></script>
    <script defer src="<{xoImgUrl 'js/alpine.min.js'}>"></script>

    <title><{$xoops_sitename}><{if !empty($xoops_pagetitle)}> - <{$xoops_pagetitle}><{/if}></title>
    <{$xoops_module_header}>
</head>

<body id="<{$xoops_dirname}>" class="min-h-screen bg-base-100 text-base-content">

<{include file="$theme_name/tpl/nav-menu.tpl"}>

<main class="container mx-auto px-4 py-6">
    <div class="flex flex-col lg:flex-row gap-6">
        <{include file="$theme_name/tpl/leftBlock.tpl"}>
        <{include file="$theme_name/tpl/content-zone.tpl"}>
        <{include file="$theme_name/tpl/rightBlock.tpl"}>
    </div>
</main>

<footer class="footer footer-center p-4 bg-base-300 text-base-content">
    <aside><{$xoops_footer}></aside>
</footer>

</body>
</html>
```

**Key design choices:**

- **`data-theme`** on `<html>` — DaisyUI reads this to apply a theme. The inline script sets it before the first paint, so there's no flash of wrong theme.
- **`bg-base-100 text-base-content`** — DaisyUI semantic colors. They adapt automatically to whichever theme is active.
- **Flex layout with `lg:flex-row`** — stacks vertically on mobile/tablet, side-by-side on desktop. Simpler than a 12-column grid and handles edge cases (empty sidebars) gracefully.
- **Alpine.js `defer`** — loaded asynchronously so it doesn't block rendering. Used for interactive components (dropdowns, mobile nav).

### Step 8 — Sidebar block templates

`tpl/leftBlock.tpl`:

```smarty
<{if $xoBlocks.canvas_left}>
<aside class="w-full lg:w-64 lg:flex-shrink-0 space-y-4">
    <{foreach item=block from=$xoBlocks.canvas_left}>
    <div class="card bg-base-200 shadow">
        <div class="card-body p-4">
            <{if $block.title}><h4 class="card-title text-base"><{$block.title}></h4><{/if}>
            <{include file="$theme_name/tpl/blockContent.tpl"}>
        </div>
    </div>
    <{/foreach}>
</aside>
<{/if}>
```

`w-64` = 16rem = 256px fixed sidebar width on large screens. `lg:flex-shrink-0` prevents it from shrinking when content is wide. Full width below the `lg` breakpoint (1024px).

`tpl/blockContent.tpl`:

```smarty
<{if !empty($xoops_isadmin)}>
<a href="<{xoAppUrl '/modules/system/admin.php?fct=blocksadmin&op=edit&bid='}><{$block.id}>"
   class="btn btn-xs btn-warning mb-2">Edit</a>
<{/if}>
<div class="block-content"><{$block.content}></div>
```

### Step 9 — Build the CSS

```bash
npm run build
```

This scans your `.tpl` files, extracts the utility classes you used, includes the DaisyUI themes you enabled, and writes everything to `css/styles.css`. Expect 100–400 KB of minified CSS depending on how many DaisyUI themes you bundled.

During development, run `npm run watch` to auto-rebuild on every file change.

### Step 10 — Activate the theme

In XOOPS admin: **System → Preferences → General Settings → Default Theme**. Pick your theme and save.

That's it. You now have a working Tailwind theme. The rest of this tutorial shows how to add theme switching and how to port an existing Bootstrap theme.

## Adding a Theme Switcher

### The dropdown

In `tpl/nav-menu.tpl`, add a DaisyUI dropdown:

```smarty
<div class="dropdown dropdown-end">
    <div tabindex="0" role="button" class="btn btn-sm btn-ghost">Theme</div>
    <ul id="mytheme-menu" tabindex="0" class="dropdown-content menu bg-base-100 rounded-box z-50 p-2 shadow max-h-96 overflow-y-auto">
        <{foreach from=$mythemeThemes item=theme}>
        <li><a href="#" data-theme-name="<{$theme.name}>"><{$theme.label}></a></li>
        <{/foreach}>
    </ul>
</div>
```

DaisyUI's `dropdown` class handles open/close via CSS (`:focus-within`). No JS needed for the dropdown itself.

### The switcher JS

```js
function mythemeSetTheme(theme) {
    document.documentElement.setAttribute('data-theme', theme);
    localStorage.setItem('mytheme-theme', theme);
}

document.addEventListener('DOMContentLoaded', function() {
    const menu = document.getElementById('mytheme-menu');
    if (menu) {
        menu.addEventListener('click', function(e) {
            const item = e.target.closest('[data-theme-name]');
            if (item) {
                e.preventDefault();
                mythemeSetTheme(item.getAttribute('data-theme-name'));
            }
        });
    }
});
```

**How it works:**
1. User clicks a theme name → JS sets `data-theme` on `<html>`
2. DaisyUI's CSS responds instantly — every element using `bg-base-100`, `text-primary`, etc. picks up the new theme's colors
3. Choice saved to `localStorage` so it persists
4. Next page load, the FOUC script reads `localStorage` and applies the theme before paint

### Dark/light toggle button

For a simple dark/light binary toggle (separate from the theme picker):

```js
function mythemeToggleMode() {
    const current = document.documentElement.getAttribute('data-theme');
    const isDark = ['dark', 'synthwave', 'dracula', 'night'].includes(current);
    mythemeSetTheme(isDark ? 'light' : 'dark');
}
```

Or use the classification from `theme_autorun.php` — store `mode: 'light'|'dark'` per theme and look it up. The xtailwind reference theme uses this approach.

## Part 2 — Porting a Bootstrap Theme to Tailwind

If you have an existing Bootstrap theme (like xswatch5) and want to convert it, this section shows the mapping.

### Key principle: component-to-utility translation

Bootstrap uses semantic classes (`btn btn-primary`, `card`, `navbar`). Tailwind uses utility classes (`inline-flex px-4 py-2 bg-blue-600 text-white rounded`). **DaisyUI bridges this gap** — it provides component classes on top of Tailwind utilities, so `btn btn-primary` works the same.

**Recommendation:** use DaisyUI for components you'd normally use Bootstrap for (buttons, cards, navbar, dropdowns, modals). Use raw Tailwind utilities for custom layouts and spacing.

### Class mapping cheat sheet

#### Layout

| Bootstrap 5 | Tailwind / DaisyUI |
|-------------|-------------------|
| `container` | `container mx-auto` |
| `row` | `flex flex-wrap -mx-4` or `grid grid-cols-12 gap-4` |
| `col-md-6` | `w-full md:w-1/2` or `md:col-span-6` |
| `col-md-3` | `w-full md:w-1/4` or `md:col-span-3` |
| `d-flex` | `flex` |
| `d-none d-md-block` | `hidden md:block` |
| `justify-content-between` | `justify-between` |
| `align-items-center` | `items-center` |

#### Spacing

| Bootstrap 5 | Tailwind |
|-------------|----------|
| `p-3` (1rem) | `p-4` (1rem) |
| `m-3` | `m-4` |
| `ms-auto` | `ms-auto` (same) |
| `me-2` | `me-2` (same) |
| `mt-4` (1.5rem) | `mt-6` (1.5rem) |
| `gap-3` | `gap-4` |

Bootstrap's spacing scale is multiples of `0.25rem` starting at `0`. Tailwind is the same scale but numbers differ: Bootstrap `p-3` = `1rem` = Tailwind `p-4`.

#### Typography

| Bootstrap 5 | Tailwind |
|-------------|----------|
| `h1`, `h2`, `h3` styles | `text-4xl font-bold`, `text-3xl font-semibold`, `text-2xl font-semibold` |
| `lead` | `text-xl font-light` |
| `text-muted` | `text-base-content/60` |
| `text-center` | `text-center` (same) |
| `text-end` | `text-end` (same) |
| `fw-bold` | `font-bold` |

#### Components (use DaisyUI)

| Bootstrap 5 | DaisyUI |
|-------------|---------|
| `btn btn-primary` | `btn btn-primary` (same!) |
| `btn-outline-primary` | `btn btn-outline btn-primary` |
| `card` | `card` (same) |
| `card-body` | `card-body` (same) |
| `card-title` | `card-title` (same) |
| `navbar navbar-expand-lg` | `navbar` + manual responsive classes |
| `dropdown-menu` | `dropdown-content menu` |
| `modal` | `modal` (same, different markup) |
| `alert alert-warning` | `alert alert-warning` (same) |
| `badge bg-primary` | `badge badge-primary` |
| `form-control` | `input input-bordered` |
| `form-select` | `select select-bordered` |
| `form-check-input` | `checkbox` or `radio` |
| `progress-bar` | `progress` |
| `nav nav-tabs` | `tabs tabs-bordered` |

### Component examples

#### Navbar

**Bootstrap:**
```html
<nav class="navbar navbar-expand-lg navbar-dark bg-primary">
    <div class="container">
        <a class="navbar-brand" href="/">Site</a>
        <button class="navbar-toggler" data-bs-toggle="collapse" data-bs-target="#nav">
            <span class="navbar-toggler-icon"></span>
        </button>
        <div class="collapse navbar-collapse" id="nav">
            <ul class="navbar-nav me-auto">
                <li class="nav-item"><a class="nav-link" href="/">Home</a></li>
            </ul>
        </div>
    </div>
</nav>
```

**DaisyUI + Alpine.js:**
```html
<div x-data="{ open: false }" class="navbar bg-primary text-primary-content">
    <div class="container mx-auto">
        <div class="flex-1">
            <a class="btn btn-ghost text-xl" href="/">Site</a>
        </div>
        <button class="btn btn-ghost lg:hidden" @click="open = !open">☰</button>
        <div class="hidden lg:flex">
            <ul class="menu menu-horizontal">
                <li><a href="/">Home</a></li>
            </ul>
        </div>
    </div>
    <div x-show="open" x-cloak class="lg:hidden absolute top-full left-0 right-0 bg-primary">
        <ul class="menu p-4">
            <li><a href="/">Home</a></li>
        </ul>
    </div>
</div>
```

Alpine.js replaces Bootstrap's `data-bs-*` collapse logic with declarative `x-data`/`x-show`/`@click` attributes. No separate JS file.

#### Modal

**Bootstrap:**
```html
<button data-bs-toggle="modal" data-bs-target="#myModal">Open</button>
<div class="modal fade" id="myModal">
    <div class="modal-dialog">
        <div class="modal-content">
            <div class="modal-header">
                <h5 class="modal-title">Title</h5>
                <button class="btn-close" data-bs-dismiss="modal"></button>
            </div>
            <div class="modal-body">Content</div>
        </div>
    </div>
</div>
```

**DaisyUI:**
```html
<button class="btn" onclick="myModal.showModal()">Open</button>
<dialog id="myModal" class="modal">
    <div class="modal-box">
        <h3 class="font-bold text-lg">Title</h3>
        <p class="py-4">Content</p>
        <div class="modal-action">
            <form method="dialog"><button class="btn">Close</button></form>
        </div>
    </div>
</dialog>
```

DaisyUI uses the native HTML `<dialog>` element — no JavaScript library required, accessible by default, and supports Escape key and backdrop-click to close for free.

#### Carousel

DaisyUI has a built-in `carousel` class but it's simpler than Bootstrap's. For parity with Bootstrap carousel (auto-play, indicators, controls), use a small JS library like [Splide](https://splidejs.com/) or [Embla](https://www.embla-carousel.com/) instead — both work well with Tailwind.

### Color mapping

Bootstrap's theme colors map to DaisyUI's semantic colors:

| Bootstrap | DaisyUI |
|-----------|---------|
| `primary` | `primary` |
| `secondary` | `secondary` |
| `success` | `success` |
| `danger` | `error` |
| `warning` | `warning` |
| `info` | `info` |
| `light` | `base-100` |
| `dark` | `base-content` |

DaisyUI also adds `accent`, `neutral`, and `base-100`/`base-200`/`base-300` for background layering. These adapt automatically per theme — no need for separate light/dark variants of your custom CSS.

### Dark mode

**Bootstrap 5.3:** `data-bs-theme="dark"` on `<html>`, `[data-bs-theme="dark"]` selectors.

**DaisyUI:** `data-theme="dark"` on `<html>`. Every DaisyUI theme is either light or dark, and the CSS variables (`--p`, `--b1`, `--bc`) adjust automatically. You don't need separate dark-mode overrides for components that use DaisyUI semantic colors.

For your own custom CSS that needs dark-specific rules:
```css
[data-theme="dark"] .my-custom-class {
    border-color: hsl(var(--b3));
}
```

### RTL

Both frameworks use CSS logical properties (`ms-`/`me-`, `ps-`/`pe-`, `text-start`/`text-end`). If your Bootstrap theme already uses these (Bootstrap 5.3+ does by default), RTL conversion is automatic — just set `dir="rtl"` on `<html>`.

### Form rendering (the tricky part)

XOOPS core generates forms via `XoopsFormRenderer`. The only available renderers are:
- `XoopsFormRendererBootstrap4`
- `XoopsFormRendererBootstrap5`
- `XoopsFormRendererLegacy`

There's no `XoopsFormRendererTailwind` in core. Your options:

1. **Use the BS5 renderer** (recommended for now) — set it in `theme_autorun.php`. DaisyUI's component classes (`btn`, `form-control`, `input-group`) overlap enough with Bootstrap's that rendered forms look acceptable.
2. **Override form templates** in your theme to output Tailwind classes directly.
3. **Contribute `XoopsFormRendererTailwind`** to XOOPS core as a long-term fix.

### Migration steps for an existing theme

1. **Set up Tailwind** alongside your existing Bootstrap theme (new directory)
2. **Port `theme.tpl` first** — get the page shell working with DaisyUI
3. **Port navigation** — use DaisyUI `navbar` + Alpine.js for mobile drawer
4. **Port sidebar blocks** — flex layout, `card` components
5. **Port content pages** — most `container`/`row`/`col` markup translates directly
6. **Port module templates one at a time** — test after each one
7. **Replace JavaScript components** — modals → `<dialog>`, collapse → Alpine `x-show`, tooltips → Tippy.js or native `title` attribute
8. **Customize theme colors** — create a custom DaisyUI theme in `tailwind.config.js` to match your brand

### What to expect

- **Class count drops significantly** — Tailwind's utilities are more granular, but each class does one thing, so markup becomes more explicit
- **CSS payload drops** — Tailwind's JIT + purging typically produces smaller CSS than Bootstrap for equivalent designs
- **Dark mode becomes trivial** — DaisyUI handles it via `data-theme`, no duplicate CSS needed
- **Component markup changes** — especially modals, dropdowns, and carousels. Test every interactive element
- **Forms may look slightly off** until you contribute a Tailwind form renderer or override the rendered HTML

## Common Gotchas

**1. Classes not appearing in output CSS** — Tailwind only includes classes it finds in files listed in `content`. If you build classes dynamically in PHP or JavaScript (e.g., `'bg-' + color`), add them to a safelist:
```js
// tailwind.config.js
content: { files: [...], extract: {...} },
safelist: ['bg-primary', 'bg-secondary', 'bg-success', ...],
```

**2. CSS variable colors not applying** — DaisyUI uses HSL CSS variables. Use them as `hsl(var(--p))` not `var(--p)`.

**3. Dropdown stays open after click** — DaisyUI dropdowns close on blur. Wrap menu items in `<label tabindex="0">` or call `document.activeElement.blur()` after selection.

**4. Forms inside XOOPS modules look unstyled** — core modules render forms with hard-coded Bootstrap classes via the form renderer. Either switch the renderer or add CSS overrides.

**5. `prose` class breaks navigation menus** — Typography's `prose` is for article text. Use a custom `.block-content` class for menu-style blocks (see input.css example above).

**6. Alpine.js flash of visible content** — elements with `x-show="false"` briefly appear before Alpine loads. Add `x-cloak` attribute and `[x-cloak] { display: none !important; }` to your CSS.

**7. Build times getting slow** — Tailwind JIT is fast, but if you enable all 34 DaisyUI themes, CSS generation takes longer. Disable themes you don't need.

**8. Changes don't appear after editing templates** — run `npm run build` or use `npm run watch`. Tailwind needs to regenerate CSS whenever you add new classes.
